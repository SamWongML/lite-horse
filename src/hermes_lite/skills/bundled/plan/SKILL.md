# plan

When a task has any meaningful complexity, slow down and write the plan before
acting. The plan is a working document — not a deliverable. Its purpose is to
make assumptions explicit, surface decisions early, and give you something to
re-orient against if you get lost.

## When to use

- A task spans 3+ logical steps.
- The request is ambiguous and you're guessing at what "done" looks like.
- Multiple files or systems will change together.
- You catch yourself starting work without a clear next step.

## How

1. Create or open `plan.md` in the working directory.
2. Write a short brief: the goal in one sentence, the constraints, the success
   check.
3. List the steps as a checklist. One bullet per step. Each step should be
   small enough to verify on its own.
4. Note open questions explicitly — don't bury them.
5. Execute one step at a time, ticking boxes as you go.
6. If reality diverges from the plan, edit the plan first, then continue.

## Anti-patterns

- Writing the plan, then ignoring it.
- Padding the plan with steps you already know how to do trivially.
- Treating the plan as immutable once written.

# lite-horse v0.2 — embed in webapp + strengthen self-evolution

**Status:** ACTIVE
**Started:** 2026-04-20
**Predecessor:** [v0.1-hermes-port.md](v0.1-hermes-port.md)
**Status ledger:** [../PROGRESS.md](../PROGRESS.md)

Each phase is sized for one engineer, one PR. Check boxes as acceptance
criteria land. Do not reorder phases without coordinating — several have hard
dependencies noted under **Depends on:**.

---

## Objective

Ship lite-horse as an **embeddable Python package** consumed by a single
project-management web backend, with a stronger self-evolution loop than v0.1
(compression-as-consolidation, periodic nudges, in-use skill refinement,
offline GEPA-style evolve). Target: **≤ 2,000 runtime LOC** after drops.

## Non-goals (explicit cuts)

- Telegram / Discord / Slack / any chat platform adapter
- CLI REPL as a product surface (keep a tiny debug entry only)
- systemd unit files shipped in the repo
- Honcho dialectic user modeling (heavy; `USER.md` covers 80 %)
- Subagent spawning as a tool
- Additional model providers (OpenAI only — matches v0.1 rule)
- 40-tool zoo; webapp-specific tools come in via MCP, not builtins

## Invariants — do not regress

1. `openai-agents >= 0.14.1, < 0.15`; no Anthropic SDK anywhere
2. Python 3.11+; `uv` for env; ruff + mypy strict + pytest must stay green
3. No orphan `asyncio.create_task` — every long-lived task has a supervisor
4. Instructions are a **frozen snapshot** at session start
5. **Prompt-cache-stable block order** (see Phase 13)
6. Any string that ends up in the system prompt must pass
   `_INJECTION_PATTERNS` + `_INVISIBLE_RE` validation
7. Memory char limits (2200 / 1375) and skill size cap (15 KB) enforced
8. No MCP URL accepted from user input — config/code only
9. Evolution never breaks the user's run — all distiller failures swallowed
10. Webapp is the only consumer; no public Python API surface outside
    `lite_horse.api`

## Phase 13 — prompt-cache fix + skill-write injection defense

**Objective:** remove the two cheap-but-high-impact cliffs from v0.1.
**Why:** `Current time: {now}` at prompt block #2 invalidates the OpenAI
prompt cache every turn (every second, actually). `skill_manage.create/edit`
accepts unvalidated content that is later concatenated into the system prompt
via `_skills_index()` — classic prompt-injection sink.
**Depends on:** none. Do this first.

### Deliverables
- [x] `src/lite_horse/agent/instructions.py` — move `f"Current time: {now}"`
      from position #2 to the **last** block, after `_TOOL_GUIDANCE`. Rename
      the docstring "Cache-stable block order" comment to reflect the new order.
- [x] `src/lite_horse/skills/manage_tool.py::dispatch` — before writing SKILL.md
      on `create`/`edit`/`write_file`, reuse `_INJECTION_PATTERNS` and
      `_INVISIBLE_RE` from `memory/store.py` (extract to
      `src/lite_horse/security/validators.py` — new module, ~30 LOC — so both
      sites import it). `memory/store.py` imports from the new module; no
      behavior change for memory.
- [x] `src/lite_horse/security/__init__.py` — empty.
- [x] `tests/test_instructions_assembly.py` — assert `Current time:` appears
      AFTER `TOOL USE GUIDANCE` in the rendered prompt.
- [x] `tests/test_skill_manage_tool.py` — add cases for
      `create` / `edit` / `write_file` with injection payloads
      (`"ignore all previous instructions"`, zero-width space) returning
      `{"success": False, "error": ...}` and NOT writing the file.

### Acceptance
- [x] All new tests green; existing 105 tests still green (110 total)
- [x] `ruff`, `mypy --strict` clean
- [ ] Manual: prompt diff between turn 1 and turn 2 of the same session is
      only the bottom `Current time:` line

### Risk / notes
- The cache retention is already set (`prompt_cache_retention="24h"`). This
  is the missing piece that lets the cache actually hit.
- If SOUL, MEMORY, or USER change mid-session (they won't — frozen-snapshot
  invariant), cache still busts; this phase only removes the time cliff.

**Est. LOC:** ~50 changed, ~50 test.

## Phase 14 — `skill_view` tool + kill dead `make_skills_capability`

**Objective:** implement real Level-1 progressive disclosure so the agent can
load a SKILL.md on demand instead of carrying every skill body in the prompt.
**Why:** `instructions.py` already tells the model to call `skill_view(name)`
— but the tool does not exist. Simultaneously, `skills/source.py::make_skills_capability()`
imports `Skills` / `LocalDirLazySkillSource` but is never passed into
`Agent(...)`; the SDK's `Agent.__init__` has no `capabilities` param. Both
issues trace back to a v0.1 plan assumption that was never wired.
**Depends on:** 13 (injection defense is required — `skill_view` returns
content that lands in the model's context).

### Deliverables
- [x] `src/lite_horse/skills/view_tool.py` — new module:
      ```python
      @function_tool(name_override="skill_view",
                     description_override="Load the full SKILL.md for <name>. "
                                          "Call this after seeing the skill in "
                                          "the AVAILABLE SKILLS index and before "
                                          "doing related work.")
      async def skill_view(ctx, name: str) -> str: ...
      ```
      Returns SKILL.md text (capped at 16 KB — one char above the 15 KB skill
      size limit). Slug-validated via the same `_SLUG_RE` from
      `manage_tool.py` (extract to `skills/_slug.py`, 5 LOC). Path-traversal
      guarded. Returns JSON `{"success": False, "error": ...}` for missing
      skill or invalid name.
- [x] `src/lite_horse/agent/factory.py` — include `skill_view` in the always-on
      tools list next to `memory_tool`, `session_search`, `skill_manage`.
- [x] `src/lite_horse/skills/source.py` — delete `make_skills_capability`
      and the `Skills` / `LocalDirLazySkillSource` / `LocalDir` imports. Keep
      `skills_root()` and `sync_bundled_skills()`.
- [x] `tests/test_skill_view_tool.py` — happy path, missing skill, slug
      violation, path-traversal attempt, content cap.

### Acceptance
- [x] `_skills_index()` still lists names + descriptions only (Level 0).
- [x] Agent can call `skill_view("plan")` and receive the full bundled skill.
- [x] `grep -r make_skills_capability src/ tests/` returns nothing.

**Est. LOC:** +60 runtime / +90 test / −15 dead.

**Objective:** one import the webapp uses. Everything else is internal.
**Why:** without this, every webapp caller reaches into `factory` + `SDKSession`

breaking the consumer.
**Depends on:** 15 (CLI deletion first so naming is unambiguous).

### Deliverables
- [ ] `src/lite_horse/api.py` — single module, all public. Exports:
      ```python
      async def run_turn(
          *,
          session_key: str,          # e.g. build_session_key(platform="web", ...)
          user_text: str,
          source: str = "web",       # recorded in SessionDB
          user_id: str | None = None,
          max_turns: int | None = None,
      ) -> RunResult: ...

      async def end_session(session_key: str, *, reason: str = "user_exit") -> None: ...

      def search_sessions(query: str, *, limit: int = 20,
                          source: str | None = None) -> list[SearchHit]: ...

      class RunResult:
          final_output: str
          session_key: str
          turn_count: int
          tool_calls: int
      ```
- [ ] `src/lite_horse/core/__init__.py`, `src/lite_horse/core/session_key.py`
      (moved in 15).
- [ ] `api.py` lazily constructs **one** process-wide `SessionDB` on first

## Phase 18 — compression-as-consolidation

**Objective:** at the budget WARNING tier (90 %), distill the trajectory into
MEMORY.md before the hard stop, so durable context survives truncation.
**Why:** v0.1 only *warns*. Hermes `context_compressor.py` consolidates
knowledge into memory layers before truncation — this is the difference
between "ran out of turns" and "ran out of turns AND lost what we learned".
**Depends on:** 13 (injection defense must cover distiller-written memory;
`MemoryStore.add` already does, but verify).

### Deliverables
- [ ] `src/lite_horse/agent/consolidator.py` — new module:
      ```python
      class Consolidator:
          """Side-agent that distills the last N turns into memory entries."""
          async def run(self, *, turn_input: list[dict]) -> list[str]:
              """Returns entries the distiller wants to add. Caller applies
                 them via MemoryStore.add(), swallowing MemoryFull /
                 UnsafeMemoryContent."""
      ```
      Side-agent instructions: "Read the trajectory below. Identify ≤5 durable
      facts (user preference / project convention / environmental constant)
      worth surviving truncation. Each entry ≤ 200 chars. Return JSON array."
      `model=cfg.model`, `max_turns=3`.
- [ ] `src/lite_horse/agent/budget.py::BudgetHook` — on transition to
      `WARNING` tier, fire the consolidator ONCE per run. Exceptions
      swallowed — consolidation must not break the run (same rule as
      `EvolutionHook`).
- [ ] `tests/test_consolidator.py` — mocked Runner returns a JSON array;
      entries end up in `MemoryStore`; MemoryFull is swallowed; injection-flagged
      entries are dropped with a log line but the run continues.

### Acceptance
- [ ] Trajectory-ending test: drive the agent to 90 % budget with a fake
      model; assert `MEMORY.md` contains ≥1 new entry written by the
      consolidator before final output.

### Notes
- Only fires at the **WARNING** tier, not the CAUTION tier — we don't want
  to pay for a distiller run every time the model does some work.
- The distiller writes via `memory_tool` would double the call surface;
  direct `MemoryStore.add` is cheaper and uses the same validation path.

**Est. LOC:** +120 runtime, +90 test.

## Phase 20 — skill stats sidecar + in-use refinement

**Objective:** track usage / success / error per skill, and on failure,
propose a patch via side-agent ("skill refinement during use", the
second Hermes verb after "create").
**Why:** v0.1 only creates skills; it never improves them. Stats also feed
Phase 24's offline evolve loop — can't optimize skills you have no data on.
**Depends on:** 14 (`skill_view` is how "skill used" is observed).

### Deliverables
- [ ] Sidecar format: `~/.litehorse/skills/<name>/.stats.json`:
      ```json
      {
        "usage_count": 0,
        "success_count": 0,
        "error_count": 0,
        "last_used_at": null,
        "last_error_at": null,
        "last_error_summary": null,
        "last_optimized_at": null,
        "schema_version": 1
      }
      ```
- [ ] `src/lite_horse/skills/stats.py` — new module: `record_view(name)`,
      `record_outcome(name, ok: bool, error_summary: str | None)`,
      `read(name)`. File-locked writes (use `fcntl` on posix,
      no-op on windows — but we target linux-only for the embed).
- [ ] `src/lite_horse/skills/view_tool.py` — on successful read, call
      `record_view(name)`.
- [ ] `src/lite_horse/agent/evolution.py::EvolutionHook`:
      1. Track which skills were viewed in the current run (from
         tool-call log).
      2. On `on_end`: if trajectory ended with model-emitted error
         markers ("error", "failed", exception tracebacks in tool results)
         AND a skill was viewed, call `record_outcome(name, ok=False, ...)`.
         Otherwise `ok=True`.
      3. If outcome is failure on a viewed skill, spawn a **refiner
         side-agent** (distinct from distiller): "Given this trajectory and
         this SKILL.md, propose ONE `skill_manage(action='patch', ...)` call
         that would have avoided the failure. Conservative — if unclear, do
         nothing."
      4. Swallow all errors.
- [ ] `src/lite_horse/agent/instructions.py::_skills_index()` — when
      rendering a skill line, append a decay tag if `error_count >= 3` AND
      `success_count / max(1, usage_count) < 0.5`: `(fragile — see stats)`.
- [ ] `tests/test_skills_stats.py` — concurrent writes don't clobber;
      record_outcome updates correct fields; sidecar created on first write.
- [ ] `tests/test_evolution_hook.py` — refiner runs on failure-with-viewed-skill;
      does NOT run on success; does NOT run on failure-without-skill-use.

### Acceptance
- [ ] 100 sequential runs exercising a skill yield a sidecar with
      `usage_count == 100` and sane success / error counts.

### Notes
- Skill failure detection via string markers is heuristic; that's fine for
  a nudge signal. Phase 22's error classifier later upgrades this to a
  structured signal.

**Est. LOC:** +160 runtime, +120 test.

## Phase 21 — conditional skill activation (agentskills.io compat)

**Objective:** stop dumping every skill in the prompt. Honor
`activate_when:` frontmatter; filter skills by match against the latest user
message.
**Why:** today `_skills_index()` lists every skill unconditionally. At 20+
skills this is pure prompt bloat. agentskills.io defines a standard `trigger`
/ `activate_when` field for this.
**Depends on:** 13, 14, 20.

### Deliverables
- [ ] Extend SKILL.md frontmatter schema to include:
      ```yaml
      activate_when:
        - keywords: ["deploy", "ship", "release"]
        - file_globs: ["Dockerfile", "k8s/*.yaml"]
      category: devops
      ```
      Parse via `yaml.safe_load` on the frontmatter block.
- [ ] `src/lite_horse/skills/activation.py` — new module:
      ```python
      def filter_for_turn(
          *,
          skills_dir: Path,
          user_text: str | None,
          top_k: int = 8,
      ) -> list[SkillEntry]: ...
      ```
      Heuristic: keyword match scores 2x, presence in recent `USER.md` scores
      1x, default-always-on (no `activate_when`) scores 0.5x. Return top-K
      sorted. NO embedding model — too heavy for a per-turn call.
- [ ] `src/lite_horse/agent/instructions.py::make_instructions` — grab the
      most recent user message from `ctx.turn_input` (safely, same pattern
      as `EvolutionHook._extract_user_request`), pass to
      `activation.filter_for_turn`. Fallback to all-skills when
      extraction fails.
- [ ] Update bundled `plan/SKILL.md` and `skill-creator/SKILL.md` with
      `activate_when` examples. Keep them always-on (no frontmatter key).
- [ ] `tests/test_skill_activation.py` — keyword match, file-glob match,
      top-K cap, fallback behavior.
- [ ] `tests/test_instructions_assembly.py` — prompt for "deploy the app"
      includes the devops skill; prompt for "write a poem" does not.

### Acceptance
- [ ] A library of 25 fake skills renders only ≤ 8 in the prompt for any
      given user message.

### Notes
- Activation is best-effort: the agent can always `skill_view(name)` anything
  it knows exists. The stats sidecar from Phase 20 can also surface skills
  that have high success on the current task cluster.

**Est. LOC:** +120 runtime, +100 test.

## Phase 22 — structured error classifier

**Objective:** replace `except Exception: log.exception(...)` with a
classifier that tags failures and drives differentiated handling.
**Why:** rate-limit should retry; model refusal should surface; context
overflow should trigger consolidation (Phase 18); tool errors should feed
Phase 20's refiner. Mixing them loses all those signals.
**Depends on:** 16, 17, 18.

### Deliverables
- [ ] `src/lite_horse/agent/errors.py` — new module:
      ```python
      class ErrorKind(enum.StrEnum):
          RATE_LIMIT = "rate_limit"
          CONTEXT_OVERFLOW = "context_overflow"
          MODEL_REFUSAL = "model_refusal"
          TOOL_ERROR = "tool_error"
          NETWORK = "network"
          UNKNOWN = "unknown"

      @dataclass
      class ClassifiedError:
          kind: ErrorKind
          retryable: bool
          original: BaseException
          summary: str

      def classify(exc: BaseException) -> ClassifiedError: ...
      ```
      Dispatches on `openai.RateLimitError`, `openai.APIConnectionError`,
      `openai.BadRequestError` (context overflow substring check), SDK
      exception types, etc.
- [ ] `src/lite_horse/api.py::run_turn` — wrap `Runner.run`, on
      `RATE_LIMIT`/`NETWORK` retry up to 3x with backoff, on
      `CONTEXT_OVERFLOW` log and return a graceful error, on
      `MODEL_REFUSAL` surface the message.
- [ ] `src/lite_horse/cron/scheduler.py::make_fire` — use `classify` to
      decide whether to retry the firing on next tick or permanently
      disable the job (3 strikes on `MODEL_REFUSAL` → job disabled with
      reason stamped in JobStore).
- [ ] `tests/test_errors.py` — each `ErrorKind` dispatched from a synthetic
      exception; classifier is total (no `UNKNOWN` leak for known types).

### Acceptance
- [ ] Injected rate-limit during `run_turn` results in 1 successful return
      after 2 retries; injected model refusal surfaces on first call.

**Est. LOC:** +110 runtime, +80 test.

## Phase 24 — offline evolve pipeline (`lite_horse.evolve`)

**Objective:** the GEPA-equivalent offline loop. Reads SessionDB traces,
mutates SKILL.md candidates, gates on pytest + size + semantic similarity,
writes **proposals** for human approval — never auto-edits.
**Why:** mirrors `nousresearch/hermes-agent-self-evolution` without pulling
DSPy into the runtime. Executes as a separate script invoked by the webapp
(button in admin UI) or by the webapp's own cron.
**Depends on:** 20 (stats drive prioritization), 22 (error kinds let us
mine failure traces).

### Deliverables — module layout
- [ ] `src/lite_horse/evolve/__init__.py`
- [ ] `src/lite_horse/evolve/trace_miner.py` — reads last N days of sessions
      from `SessionDB`, extracts `(task, response, outcome)` triples where
      outcome uses the error classifier's `ErrorKind`.
- [ ] `src/lite_horse/evolve/fitness.py` — LLM-as-judge rubric (0-1 per
      sample) + length penalty + `text-embedding-3-small` cosine against the
      baseline SKILL.md.
- [ ] `src/lite_horse/evolve/constraints.py` — hard gates:
      - pytest suite passes
      - size ≤ 15 KB
      - semantic cosine ≥ 0.75 vs. baseline (catches purpose drift)
      - no `_INJECTION_PATTERNS` / `_INVISIBLE_RE` hits
      - frontmatter valid YAML, preserves `name:` and bumps `version:`
- [ ] `src/lite_horse/evolve/reflector.py` — single-step LLM reflection:
      "Given the skill text, 3-5 failure trajectories, and a rubric,
      propose ONE revised SKILL.md. Explain changes in a comment above the
      frontmatter (stripped before validation)."
      No DSPy dep — straight `openai` calls. If we want DSPy later, wrap it
      behind this interface.
- [ ] `src/lite_horse/evolve/runner.py` — CLI-free entry: `evolve(skill_name:
      str) -> EvolveResult`. Loops: mine → reflect → constraint-gate →
      write `.proposals/<skill_name>/<timestamp>.md` with metrics JSON sidecar.
- [ ] `src/lite_horse/evolve/cli.py` — `python -m lite_horse.evolve <skill>`
      shim for admin invocation. Not a `litehorse` subcommand — separate
      module keeps it out of the import graph of the webapp runtime.
- [ ] `tests/test_evolve_constraints.py` — each gate isolated.
- [ ] `tests/test_evolve_e2e.py` — end-to-end with mocked LLM + synthetic
      traces.

### Deliverables — surface in webapp
- [ ] `~/.litehorse/skills/.proposals/<skill>/<ts>.md` + `<ts>.json` with
      `{"baseline_score": ..., "candidate_score": ..., "gates": {...}}`.
- [ ] The webapp lists pending proposals and exposes approve / reject.
      Approve = move the candidate to `~/.litehorse/skills/<skill>/SKILL.md`
      and stamp `last_optimized_at` in the stats sidecar.

### Acceptance
- [ ] `python -m lite_horse.evolve plan` against a seeded SessionDB produces
      a proposal; constraint violations correctly reject candidates; the
      webapp surface renders the before/after diff.
- [ ] `import lite_horse.api` does **not** transitively import
      `lite_horse.evolve`. Verified by
      `python -c "import sys, lite_horse.api; assert 'lite_horse.evolve' not in sys.modules"`.

### Notes
- Do **not** add DSPy or GEPA as runtime deps. If we want them, they go in
  an optional `[project.optional-dependencies] evolve = [...]` extra.
- Baseline approach is single-pass LLM reflection; upgrade to GEPA later if
  the single-pass isn't finding useful mutations after real usage.

**Est. LOC:** +350 (separate module) / +180 test.

# lite-horse v0.3 — scripted subcommand CLI (`litehorse`)

**Status:** ACTIVE
**Started:** 2026-04-23
**Predecessor:** [v0.2-embed-and-evolve.md](v0.2-embed-and-evolve.md)
**Status ledger:** [../PROGRESS.md](../PROGRESS.md)

---

## Objective

Ship a **sophisticated, scripted, subcommand-style** CLI at `litehorse`.
Developer-facing. Composable. No TUI, no interactive REPL default, no
chat-platform adapter. Every command does one thing and exits.

This reverses v0.2's "no CLI as product surface" stance. Reasoning:

1. The embed-in-webapp story is intact; `lite_horse.api` is unchanged.
   The CLI is a *second* consumer of the same internal modules, not a
   replacement.
2. Running `litehorse skills evolve <name>`, `litehorse sessions search ...`,
   `litehorse cron list` from CI / cron / developer shell is strictly better
   than `python -c "from lite_horse... ; ..."` one-liners.
3. Hermes (upstream) and openclaw both ship rich CLIs; shipping parity on
   this surface makes our runtime easier to operate and integrate.

`litehorse-debug` is deleted at the end of this plan — fully replaced.

## Non-goals (explicit cuts)

- Interactive REPL / TUI. (If we ever want this, it is a separate package.)
- Any chat-platform adapter (Telegram, Slack, etc. — permanently out.)
- Daemon / gateway / RPC protocol. The CLI runs the agent in-process and
  exits. No persistent process, no socket, no RPC.
- Interactive config wizards (`configure`, `onboard`). Developers use
  `litehorse config edit` + documented env vars.
- Mid-session slash commands (`/model`, `/compress`). We are single-turn.

## Hard contract (binding for every implementation subagent)

Every implementation phase below **must** conform to these. A subagent that
cannot honor one of these points should stop and escalate, not invent a
workaround.

### Entry point
- `[project.scripts] litehorse = "lite_horse.cli.app:main"` in `pyproject.toml`.
- `litehorse-debug` script is removed in Phase 30. Until then, both coexist.

### Package layout
```
src/lite_horse/cli/
    __init__.py
    app.py                 # Typer() root app; `main()` exported
    _async.py              # @arun decorator (asyncio.run wrapper)
    _output.py             # human vs JSON emit helpers
    _signals.py            # SIGINT/SIGTERM cooperative stop
    _settings.py           # pydantic-settings cascade
    exit_codes.py          # IntEnum of documented exit codes
    commands/
        __init__.py
        run.py             # `litehorse run ...`
        sessions.py        # `litehorse sessions ...`
        skills.py          # `litehorse skills ...`  (incl. proposals + evolve)
        cron.py            # `litehorse cron ...`
        memory.py          # `litehorse memory ...`
        config.py          # `litehorse config ...`
        doctor.py          # `litehorse doctor`
        logs.py            # `litehorse logs ...`
        version.py         # `litehorse version`
```

### Library stack (additions to pyproject)
- `typer >= 0.12, < 1.0` (adds `click`, `rich`, `typing-extensions`)
- `pydantic-settings >= 2.2, < 3`

No other runtime deps added. `rich` comes transitively via `typer[all]` is
**not** acceptable — pin the minimal extras. The import surface must stay
under what we already have when `rich` is merged.

### Async handler pattern
Every async Typer command body uses the `@arun` decorator from
`_async.py`. No raw `asyncio.run()` in command bodies. No custom loops.

```python
from lite_horse.cli._async import arun

@app.command()
@arun
async def run(prompt: str, session: str = "...") -> None:
    ...
```

### Output discipline
- **Data → stdout**, logs + progress → stderr. Always. No exceptions.
- `--json` is a `bool = False` flag on every command that produces output.
  When true: emit **NDJSON** (one JSON object per line) to stdout, nothing
  else. Progress suppressed.
- Human mode uses `rich.Console(stderr=True)` so piping auto-strips ANSI.
- JSON record shape is one of:
  ```
  {"kind": "result", "data": {...}}
  {"kind": "item",   "data": {...}}      # for list streams
  {"kind": "progress","stage": "...", "pct": 0.4}   # optional, only if --progress
  {"kind": "error",  "code": 3, "message": "..."}
  ```
- `--help` output goes to **stdout** (explicit exception; tooling expects it).

### Exit-code policy
Defined once in `exit_codes.py` as `IntEnum`:
- `0` OK
- `1` generic unhandled error
- `2` usage error (bad flag, missing arg) — Typer default
- `3` agent turn failed (error-kind other than rate-limit/network after retries)
- `4` config / auth error (missing `OPENAI_API_KEY`, invalid config.yaml)
- `5` not found (session/skill/job/proposal target absent)
- `6` gate failure (evolve proposal rejected by a gate — not an *error*, a signal)
- `7` IO / state error (db locked, disk full, perms)
- `130` interrupted (SIGINT). Must print nothing extra in JSON mode.

Every command docstring includes an "Exit codes" section in `--help`.

### Config cascade
`pydantic-settings`-backed `CLISettings` in `_settings.py`. Precedence
(highest wins): **CLI flag > env var > `~/.litehorse/config.yaml` > default**.

`customise_sources` is set explicitly. Env vars are prefixed
`LITEHORSE_` (e.g. `LITEHORSE_MODEL`). Flags feed into the settings via
`Settings.model_validate({...})` at the start of each command; do not mutate
the process environment.

### Signal handling
`_signals.py` exports an async context manager:

```python
async with graceful_stop() as stop_event:
    # on SIGINT/SIGTERM: stop_event.set(); current task gets a chance to finish
    await agent.run_turn(..., stop_event=stop_event)   # when supported
```

First signal: cooperative. Second signal: hard cancel. Exit code `130`
either way. Partial state always persists in a `finally` block.

### Fast-path help
`app.py` short-circuits `--help` / `--version` *before* importing
`lite_horse.api`, `openai`, or `agents`. Validated by a unit test that
monkeypatches `sys.modules` and asserts the heavy modules don't get pulled.

### Testing
- `typer.testing.CliRunner(mix_stderr=False)` for every command.
- One file per command group: `tests/cli/test_run.py`,
  `tests/cli/test_sessions.py`, etc. New dir `tests/cli/` with its own
  `__init__.py` and `conftest.py`.
- Hermetic: reuse the existing `litehorse_home` fixture (temp
  `LITEHORSE_HOME`).
- Assert on structured JSON (`result.stdout`) when testing `--json` paths;
  on stderr pattern for human-mode banners.

### Docs
One new doc: `docs/CLI.md`. Auto-invoked sections:
- Command table
- Exit codes
- Output formats
- Config cascade
- Signal handling
- Completion install

│
├── skills
│   ├── list                     # --json   shows name, version, stats sidecar fields
│   ├── show NAME                # --json   rendered SKILL.md; with --json dumps frontmatter+body
│   ├── evolve NAME              # --days N --json   invokes lite_horse.evolve.evolve
│   └── proposals
│       ├── list                 # --skill NAME --json
│       ├── show PATH            # --json
│       ├── approve PATH         # --force --json   copies to live skill + mark_optimized
│       └── reject PATH          # --json   deletes pair
│
├── cron
│   ├── list                     # --json
│   ├── add                      # --schedule "0 9 * * *" --prompt TEXT
│   │                            # --delivery webhook|log --url URL --json

│   ├── run-once ID              # --json   fires now, exits

│   └── clear                    # --which memory|user|both --dry-run

│   ├── path                     # absolute path of config.yaml

├── doctor                       # --json   checks config, env, DB, skills dir perms,
│                                #          OPENAI_API_KEY reachability
│
├── logs

│   └── path                     # absolute path of current log dir

### Phase 27 — `run` + `sessions`

**Objective:** the two most-used verbs. After this phase a developer can
drive the agent from the shell end-to-end.

**Deliverables**
- `commands/run.py`, `commands/sessions.py`.
- `run`: wraps `lite_horse.api.run_turn`. Human mode streams progress to
  stderr (spinner), final reply to stdout. JSON mode emits one
  `{"kind":"result","data":RunResult_as_dict}` on exit.
- `sessions list`: newest-first, respects `--limit` (default 20), `--source`.
  Human: table via Rich. JSON: NDJSON item stream then a single
  `{"kind":"result","data":{"count":N}}` terminator.
- `sessions show SESSION_KEY`: prints messages chronological. Respects
  `--limit`. JSON: NDJSON items.
- `sessions search QUERY`: wraps `api.search_sessions`. Respects
  `--limit`, `--source`, `--role`. NDJSON items + terminator.
- `sessions end SESSION_KEY [--reason]`: calls `api.end_session`. Exit 5
  if session unknown.
- `sessions cleanup --older-than-days N [--dry-run]`: deletes sessions
  (and their messages) older than N days. `--dry-run` emits what would
  be deleted, deletes nothing. Exit 0 with counts.
- Tests: `tests/cli/test_run.py`, `tests/cli/test_sessions.py`. Mock the
  Agent / OpenAI call so tests stay hermetic.

**Acceptance**
- [ ] `litehorse run "hello" --session test:dm:1 --json` produces
      a single `{"kind":"result",...}` line with `final_output`.
- [ ] `litehorse sessions search "<query>" --json | wc -l` matches hit count + 1.
- [ ] `sessions cleanup --dry-run` never mutates DB (post-condition check
      in test).
- [ ] SIGINT during `run` produces exit 130 *and* `ended_at` is stamped
      on the partial session row (verified by a signal test).

**Est. LOC:** +400 runtime / +350 test.

### Phase 28 — `skills` + `skills proposals` + `skills evolve`

**Objective:** full skill lifecycle from the CLI. Consolidates the Phase 24
offline evolve plumbing into a real operator workflow.

**Deliverables**
- `commands/skills.py` with a Typer sub-app.
- `skills list`: directory walk of `~/.litehorse/skills/`, one row per skill
  with `name`, `version`, `usage_count`, `success_count`, `error_count`,
  `last_used_at`, `last_optimized_at`. JSON: NDJSON items.
- `skills show NAME`: prints SKILL.md. With `--json`, emits
  `{"kind":"result","data":{"frontmatter":{...},"body":"..."}}`.
- `skills evolve NAME [--days N]`: calls `lite_horse.evolve.evolve`. Exit 0
  on approve, 6 on gate rejection, 5 if skill missing. JSON: one
  `{"kind":"result","data":asdict(EvolveResult)}`.
- `skills proposals list [--skill NAME]`: directory walk of
  `skills/.proposals/<skill>/`. NDJSON items with paths + sidecar summary.
- `skills proposals show PATH`: prints proposal .md + its sidecar
  (`--json` merges both).
- `skills proposals approve PATH [--force]`:
  - validates path is under `.proposals/<skill>/...`
  - re-runs the constraint gates from `lite_horse.evolve.constraints` on the
    proposal content one more time (defense in depth against stale files);
    aborts with exit 6 if any gate fails unless `--force`
  - copies proposal `.md` over the live `SKILL.md`
  - calls `lite_horse.skills.stats.mark_optimized(name)`
  - deletes the proposal `.md` + sidecar `.json` after success
- `skills proposals reject PATH`: deletes the pair, exit 0.
- Tests: `tests/cli/test_skills.py`, `tests/cli/test_skills_proposals.py`.
  Stub the reflector/judge/embedder/pytest_runner so `skills evolve` is
  hermetic (same pattern as `test_evolve_e2e.py`).

**Acceptance**
- [ ] `skills evolve <skill>` seeded via the test's `_good_candidate` stub
      writes a proposal and `skills proposals approve` replaces the live
      `SKILL.md` + bumps stats sidecar `last_optimized_at`.
- [ ] `skills proposals approve` on a tampered file (size > 15 KB inserted
      after the fact) rejects with exit 6; `--force` bypasses and test
      asserts the warning went to stderr.
- [ ] Path escape (`skills proposals show ../etc/passwd`) refused with
      exit 2 and no filesystem read. Subagent **must** add this test.

**Est. LOC:** +400 runtime / +350 test.

### Phase 29 — `cron` + `memory`

**Objective:** replace the `python -c "from lite_horse.cron.scheduler..."`
one-liner, expose cron maintenance, expose memory inspection.

**Deliverables**
- `commands/cron.py`:
  - `list`: reads `JobStore`, NDJSON item stream with per-job state
  - `add --schedule --prompt --delivery {webhook|log} [--url]`: validates
    schedule syntax *before* writing; exit 2 on bad syntax.
  - `enable ID`, `disable ID` (the latter writes `disabled_reason="manual"`).
  - `remove ID`: exit 5 if ID unknown.
  - `run-once ID`: fires the job inline (without waiting for APScheduler
    trigger); exits when done. Uses the same `run_turn` + delivery path.
  - `scheduler`: `run_scheduler_blocking()`. Installs SIGINT/SIGTERM
    handlers that call APScheduler's `shutdown(wait=True)`. Logs to stderr.
- `commands/memory.py`:
  - `show --which memory|user|both`: prints files. JSON: merged dict.
  - `clear --which memory|user|both [--dry-run]`: zeros the selected file(s).
    Dry-run prints what would be deleted and exits 0 without mutating.
- Tests: `tests/cli/test_cron.py`, `tests/cli/test_memory.py`.
- Update `docs/EMBEDDING.md`: the "Running it" snippet becomes
  `litehorse cron scheduler`; the legacy `python -c "..."` invocation is
  documented as deprecated but still supported.

**Acceptance**
- [ ] `cron add --schedule "bogus" --prompt x --delivery log` exits 2 with
      stderr message naming the bad field.
- [ ] `cron run-once ID` with a stub webhook posts to a `pytest_httpserver`
      fixture and verifies signature (reuse `tests/test_cron_delivery.py`
      HMAC check).
- [ ] `memory clear --which both --dry-run` leaves files unchanged.
- [ ] `cron scheduler` SIGTERM test: spawn in a subprocess (this is the
      one case where subprocess is warranted), send SIGTERM, assert exit
      code 130 and APScheduler `shutdown` was called (observable via a
      sentinel file written by a monkeypatched job).

**Est. LOC:** +400 runtime / +350 test.

### Phase 30 — structured logs + `logs` + `debug share` + removal

**Objective:** observability. Structured log sink. `litehorse logs`
surface. Remove `litehorse-debug` and `_debug_cli.py`. Final docs.

**Deliverables**
- `src/lite_horse/observability/logs.py` (new module; not under `cli/`):
  - Installs a logging handler that writes NDJSON records to
    `~/.litehorse/logs/YYYYMMDD.log`.
  - One record per log line: `{"ts":...,"level":...,"logger":...,"msg":...,
    "extra":{...}}`.
  - Rotation: keep last 7 days, gzip older. Atomic.
  - **Opt-in** via `LITEHORSE_STRUCTURED_LOGS=1` env or `observability.structured_logs: true`
    in `config.yaml`. Default OFF so we don't change runtime I/O for the
    webapp without notice.
  - Installed by `lite_horse.cli.app.main()` when the setting is on, and
    by `lite_horse.api._ensure_ready()` when the setting is on. Single
    installation guard.
- `commands/logs.py`:
  - `logs tail [--follow] [--lines N]`: human mode pretty-prints;
    `--json` raw NDJSON passthrough.
  - `logs path`: prints absolute path of current log file.
- `doctor` extension: reports structured-log state (on/off, path, size,
  last write age).
- `commands/debug.py` (new command group):
  - `debug share [--output PATH]`: collects last 500 log lines, config
    (redacted — API keys → `***`), skill list, job list, session count,
    schema version, package version. Writes a tarball.
    No upload. Prints path.
- **Delete** `src/lite_horse/_debug_cli.py`.
- **Remove** `litehorse-debug` from `pyproject.toml`.
- `docs/CLI.md` completed — every command documented, every exit code
  explained, completion install snippets for bash/zsh/fish.
- Update `README.md`: CLI is now the primary developer-facing surface.
- Update `docs/EMBEDDING.md`: webapp integration unchanged; add a short
  section on structured logs for webapp-side tailing.

**Acceptance**
- [ ] With `LITEHORSE_STRUCTURED_LOGS=1`, a `run` invocation writes at
      least one NDJSON record to `~/.litehorse/logs/`.
- [ ] With the flag off, no file in `~/.litehorse/logs/` is created.
- [ ] `logs tail --lines 5 --json` emits exactly 5 valid JSON lines.
- [ ] `debug share --output /tmp/bundle.tgz` creates a tar that contains
      `config.yaml` with the `OPENAI_API_KEY` line redacted to `***`.
- [ ] `which litehorse-debug` fails after install (entry point gone).
- [ ] `uv run pytest -q` still green. Full-repo `ruff`/`mypy` still clean.

**Est. LOC:** +450 runtime / +250 test.

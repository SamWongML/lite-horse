# lite-horse v0.4 — cloud multi-tenant backend with admin injection

**Status:** SHIPPED
**Started:** 2026-04-26
**Shipped:** 2026-04-30
**Predecessor:** [v0.3-cli-entrypoint.md](v0.3-cli-entrypoint.md)
**Status ledger:** [../PROGRESS.md](../PROGRESS.md)

---

## Objective

Turn lite-horse from a single-user, embedded Python library + REPL into a
**multi-tenant HTTP backend** deployed on AWS ECS behind ALB, consumed by
the existing webapp's chat tab + per-user settings page. Each webapp user
owns a personal assistant whose memory.md, user.md, skills, MCP servers,
commands, and cron jobs persist in RDS Postgres. Admins push **official**
elements (instructions, skills, commands, MCP servers, cron jobs) that
layer over each user's customisation with mandatory / opt-out semantics.

The CLI from v0.3 stays as a developer convenience but is **not deployed**.
The webapp is the only product surface in v0.4.

This is a substantial deployment shape change (from single-process local
state to stateless ECS tasks against shared RDS / Redis / S3) and a major
new feature (layered config with admin governance). It is also the
parity-with-popular-repos work: skills hub-style management, multi-provider
support, cost metering, MCP per-tenant pools, and structured observability.

## Non-goals (explicit cuts)

- **External agent runners (codex CLI, opencode CLI).** Out. The single
  in-process agent loop is the only execution path.
- **Messaging-channel gateways** (Telegram, Slack, Discord, etc.). Out;
  the existing webapp is the only channel.
- **Voice / wake-word / canvas / companion apps.** Out.
- **Full sandboxed shell execution.** No shell tool is added in v0.4;
  revisit when shell tools land.
- **Vector / pgvector retrieval.** Heuristic skill activation + tsvector
  FTS are sufficient for v0.4. pgvector add-on is a v0.5 candidate.
- **Subagent spawning** (`agent_spawn` tool). v0.5 candidate.
- **REPL feature work.** v0.3 ships the REPL; v0.4 freezes it. The REPL
  must keep building locally against the new repository layer (via a
  SQLite-backed `LocalSessionRepo` impl behind the same Protocol used by
  Postgres) so dev workflows survive.

## Research synthesis (binding — do not re-research)

Three tracks of research delivered 2026-04-26. All sources logged in
session transcripts.

### From `github.com/NousResearch/hermes-agent` (community-praised features)

- **Skills hub at `agentskills.io`** — open shareable schema. We mirror
  the SKILL.md frontmatter shape so future import is trivial.
- **3-layer memory**: FTS5 session search + LLM summarisation + Honcho
  user model. We have layers 1 + 2 (session_search + Consolidator);
  layer 3 (long-running user model) is approximated by `user.md`
  growing under admin-allowed bounds.
- **Provider-agnostic.** Drives v0.4's `ModelProvider` adapter (OpenAI +
  Anthropic at minimum, BYO key per user).
- **40+ first-party tools.** v0.4 does not bundle a fs/shell battery;
  it does land **bundled GitHub tools** (Phase 37) since the product
  domain is software issue fixing.
- **Skill auto-creation.** We already have offline evolution with hard
  gates — strictly safer; no autonomous-merge.

### From `github.com/openclaw/openclaw` (community-praised features)

- **Bundled / managed / workspace skill tiers.** Adopted verbatim as
  `bundled` / `official` / `user` scopes — central concept of v0.4.
- **MCP registry** as a first-class entity. v0.4 makes `mcp_servers` a
  layered DB-backed table per scope.
- **Sandbox modes**, `openclaw doctor`, **DM pairing** — relevant
  patterns. v0.4 does not implement sandboxing (no shell tool), keeps
  `litehorse doctor`, and replaces DM-pairing with JWT-from-webapp.
- **Heartbeat cron.** Already present (`cron/`); v0.4 splits scheduling
  out of the API process.

### From multi-tenant SaaS architecture references

- **Tenant ID consistently propagated across data, app, and UI.**
  Drives the `app.user_id` GUC + RLS strategy.
- **Configuration over customisation.** Drives the layered-config model
  (no per-tenant code forks).
- **Event-driven config invalidation.** Drives the Redis pub/sub cache
  invalidation on admin writes.
- **Single KMS key per tenant cost-conscious model.** We use **one app
  KMS key with `EncryptionContext={"user_id": ...}`** rather than
  per-tenant keys; cost-zero for tenant onboarding.

### From AWS-native deployment references (binding choices)

- **SQLAlchemy 2.0 async + asyncpg driver.** Best perf for async FastAPI;
  ORM doubles as Alembic source of truth.
- **MinIO** for local S3 testability (boto3-compatible); **moto** for
  unit tests.
- **`aws-secretsmanager-caching-python`** for in-memory Secrets Manager
  caching with TTL.
- **KMS envelope encryption + DataKey caching via AWS Encryption SDK**
  for per-user BYO keys (avoids the $0.40/secret/month tax of putting
  every user key in Secrets Manager).
- **In-process `asyncpg.Pool` first; RDS Proxy only when crossing ~50
  ECS tasks.**
- **APScheduler with `SQLAlchemyJobStore` in a separate ECS service**
  for cron — simpler than EventBridge at this scale.

## Hard contract (binding for every implementation subagent)

Any subagent that cannot honor one of these must **stop and escalate**.

### Tenancy & isolation

- Every state-bearing module reads `user_id` from a `contextvars.ContextVar`
  set by the JWT middleware. **No module accepts an implicit `current_user`
  global.**
- Every Postgres connection sets `app.user_id` via `SELECT
  set_config('app.user_id', $1, true)` on checkout. RLS policies enforce
  tenant isolation as defence-in-depth.
- **No code outside `src/lite_horse/storage/` may import `boto3`,
  `aiobotocore`, `redis`, or `aws_encryption_sdk` directly.** All cloud
  primitives are accessed through the storage protocols.
- **No code outside `src/lite_horse/repositories/` may issue raw SQL.**
  Repositories are the only callers of `AsyncSession`.

### Storage abstraction layers

- `BlobStore`, `SecretsProvider`, `Kms`, `SessionLock` are protocols with
  ≥2 impls each: production (S3 / Secrets Manager / AWS KMS / Redis) and
  local (Local FS / `.env` / Fernet / in-memory).
- Selection is by `LITEHORSE_ENV` env var (`local` / `dev` / `prod`).
  Never by feature flag, never per-call.
- Tests default to the `local` impls. A separate integration suite
  exercises MinIO / moto / fake KMS.

### Layered config resolution

- The three scopes are exactly: `bundled` (in image, read-only),
  `official` (admin-injected, DB), `user` (per-user, DB).
- Resolution rule (frozen): `effective = bundled ∪ official ∪ user_for(this_user)`,
  with mandatory officials always included, opt-outs filtering non-mandatory
  officials, user-scope shadowing official only when official is
  non-mandatory. Slug collisions resolved in this order.
- `effective-config` is computed per turn, cached in Redis with
  60 s TTL, invalidated by Redis pub/sub on admin writes. **Never cache
  longer than 60 s.**

### Versioning

- Every official entity is **immutable per version**: edits create a new
  row with incremented `version`, the previous row keeps `is_current=false`.
- Per-scope partial unique indexes enforce one current row per slug.
- Rollback flips `is_current` back to a prior row, never deletes history.
- User entities are mutable in place (no version history at this stage).

### Auth model

- JWT verified against issuer JWKS (cached 1 h) on every request.
- Required claims: `sub`, `role` (`'user'` or `'admin'`), `exp`, `iss`.
- `role='admin'` is the only gate for `/v1/admin/*`. Returns 403 otherwise.
- Every admin write hits `audit_log` with actor, action, before/after diff.

### Async discipline

- All HTTP routes are `async def`. All repository methods are `async`.
  All tool functions called by the agent loop remain whatever shape the
  agents SDK requires.
- Per-request `AsyncSession` is opened by a FastAPI dependency, closed
  in `finally`. **No long-lived sessions.**
- The `AsyncEngine` is a single module-level singleton per process, sized
  `pool_size=5, max_overflow=5, pool_pre_ping=True, pool_recycle=1800`.

### Encryption at rest

- Two columns get KMS-envelope-encrypted bytes:
  `users.byo_provider_key_*` and `mcp_servers.auth_value_*`.
- Encryption uses **one app KMS key** with
  `EncryptionContext={"user_id": str(user_id)}` for binding. Decrypt
  fails fast on context mismatch.
- DataKey caching via AWS Encryption SDK; cache TTL 5 min, max 1000
  entries per process.

### Local-testable end-to-end

- `docker-compose up` brings: Postgres 16, Redis 7, MinIO with seeded
  buckets, the `api` service, and the `scheduler` service.
- `make dev` runs migrations, seeds bundled instructions/commands, and
  seeds an admin user matching a local JWT signing key.
- The full integration test suite runs against this stack in CI.

### Migrations

- Alembic only. The hand-rolled `_migrate` ladder in `sessions/db.py` is
  deleted, not ported.
- Every migration runs under `pg_advisory_lock(0xL1TE_HORSE)` so
  concurrent ECS task starts serialise.
- Migrations execute as a **one-shot ECS task** in the deploy pipeline
  before service rolling update. Containers do not migrate on start.

### Observability

- Structured JSON logs to stdout via `structlog`. Default-on in
  `LITEHORSE_ENV=prod`. Required fields per log: `request_id`, `user_id`
  (when known), `session_key` (when known), `turn_id` (when known),
  `tool_name?`, `model?`, `latency_ms`, `tokens_in?`, `tokens_out?`,
  `error_kind?`.
- OpenTelemetry SDK wired with OTLP exporter; spans on `run_turn`, every
  `@function_tool`, every MCP call, every DB transaction. Production
  ECS tasks ship traces to ADOT sidecar → X-Ray.
- Metrics via EMF JSON inside log lines (no separate metrics push).
  Required series: `turns_total`, `tool_calls_total`, `errors_total`,
  `mcp_call_duration_ms`, `db_query_duration_ms`, `tokens_total`,
  `cost_usd_micro`, `cron_fires_total`.

### What deletes from v0.3 code

- `src/lite_horse/sessions/db.py` (481 LOC) — replaced by
  `repositories/session_repo.py` + `repositories/message_repo.py`.
- `_Writer` retry loop, `threading.local` connection cache,
  `wal_checkpoint`, `_sanitize_fts5_query`, inline `_migrate` ladder —
  all gone.
- `lite_horse.constants.litehorse_home()` references in non-CLI modules.
  CLI keeps its own dev-mode home for the local SQLite-backed repo impl.
- Config keys `sandbox.enabled` and `gateway.telegram.*` (latter still
  ignored from v0.2; now removed from the schema).

### What stays untouched

- `agent/factory.build_agent` algorithm (its inputs change source: from
  config + filesystem to repositories).
- `agent/instructions`, `agent/budget`, `agent/evolution`,
  `agent/consolidator`, `agent/errors`.
- `core/permission`, `core/session_key`, `security/validators`.
- `evolve/` algorithm; only `trace_miner` (reads PG instead of SQLite)
  and proposal sink (writes `skill_proposals` instead of disk) change.
- `cron/jobs.py` model, `cron/manage_tool.py`, `cron/delivery.py`.
- The REPL (`cli/repl/*`) — kept as a developer aid, configured to use
  the local SQLite-backed `LocalSessionRepo` impl by default.

## Locked architecture

```
Existing webapp ──HTTPS──> ALB ──┬──> ECS service "api"      (FastAPI, autoscale)
                                 │       │ asyncpg pool ──> RDS Postgres (Multi-AZ, gp3)
                                 │       │ redis-py     ──> ElastiCache Redis
                                 │       │ aiobotocore  ──> S3 (or MinIO locally)
                                 │       │ aws-sm-cache ──> Secrets Manager
                                 │       │ aws-enc-sdk  ──> AWS KMS
                                 │       │ httpx        ──> per-user MCP HTTP servers
                                 │       └ openai / anthropic ──> LLM providers
                                 │
                                 ├──> ECS service "scheduler" (1 task)
                                 │       └ APScheduler + SQLAlchemyJobStore on RDS
                                 │       └ on fire ──> SQS ──> "worker" service
                                 │
                                 └──> ECS service "worker"   (autoscale on queue depth)
                                         └ same code path as api; runs cron + evolve
```

## Locked data model (DDL)

Names use `snake_case`. PKs are `UUID` for tenant entities, `BIGINT
GENERATED ALWAYS AS IDENTITY` for high-volume append-only tables.

```sql
-- ============ Identity ============
CREATE TABLE users (
  id                   UUID PRIMARY KEY,
  external_id          TEXT NOT NULL UNIQUE,
  role                 TEXT NOT NULL DEFAULT 'user' CHECK (role IN ('user','admin')),
  created_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  default_model        TEXT,
  permission_mode      TEXT NOT NULL DEFAULT 'auto' CHECK (permission_mode IN ('auto','ask','ro')),
  byo_provider_key_ct  BYTEA,
  byo_provider_key_dk  BYTEA
);

-- ============ Layered config (pattern repeated per entity) ============
CREATE TABLE skills (
  id              UUID PRIMARY KEY,
  scope           TEXT NOT NULL CHECK (scope IN ('official','user')),
  user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
  slug            TEXT NOT NULL,
  version         INT NOT NULL DEFAULT 1,
  is_current      BOOLEAN NOT NULL DEFAULT TRUE,
  mandatory       BOOLEAN NOT NULL DEFAULT FALSE,
  enabled_default BOOLEAN NOT NULL DEFAULT TRUE,
  frontmatter     JSONB NOT NULL,
  body            TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by      UUID REFERENCES users(id),
  CHECK ((scope='user' AND user_id IS NOT NULL) OR (scope='official' AND user_id IS NULL))
);
CREATE UNIQUE INDEX skills_official_slug_current ON skills (slug)         WHERE scope='official' AND is_current;
CREATE UNIQUE INDEX skills_user_slug_current     ON skills (user_id,slug) WHERE scope='user'     AND is_current;

CREATE TABLE instructions (
  id          UUID PRIMARY KEY,
  scope       TEXT NOT NULL CHECK (scope IN ('official','user')),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  slug        TEXT NOT NULL,
  version     INT NOT NULL DEFAULT 1,
  is_current  BOOLEAN NOT NULL DEFAULT TRUE,
  mandatory   BOOLEAN NOT NULL DEFAULT FALSE,
  priority    INT NOT NULL DEFAULT 100,
  body        TEXT NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by  UUID REFERENCES users(id)
);
CREATE UNIQUE INDEX instructions_official_slug_current ON instructions (slug)         WHERE scope='official' AND is_current;
CREATE UNIQUE INDEX instructions_user_slug_current     ON instructions (user_id,slug) WHERE scope='user'     AND is_current;

CREATE TABLE commands (
  id          UUID PRIMARY KEY,
  scope       TEXT NOT NULL CHECK (scope IN ('official','user')),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  slug        TEXT NOT NULL,
  version     INT NOT NULL DEFAULT 1,
  is_current  BOOLEAN NOT NULL DEFAULT TRUE,
  mandatory   BOOLEAN NOT NULL DEFAULT FALSE,
  description TEXT,
  prompt_tpl  TEXT NOT NULL,
  arg_schema  JSONB,
  bind_skills TEXT[],
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by  UUID REFERENCES users(id)
);
CREATE UNIQUE INDEX commands_official_slug_current ON commands (slug)         WHERE scope='official' AND is_current;
CREATE UNIQUE INDEX commands_user_slug_current     ON commands (user_id,slug) WHERE scope='user'     AND is_current;

CREATE TABLE mcp_servers (
  id               UUID PRIMARY KEY,
  scope            TEXT NOT NULL CHECK (scope IN ('official','user')),
  user_id          UUID REFERENCES users(id) ON DELETE CASCADE,
  slug             TEXT NOT NULL,
  url              TEXT NOT NULL,
  auth_header      TEXT,
  auth_value_ct    BYTEA,
  auth_value_dk    BYTEA,
  cache_tools_list BOOLEAN NOT NULL DEFAULT TRUE,
  enabled          BOOLEAN NOT NULL DEFAULT TRUE,
  mandatory        BOOLEAN NOT NULL DEFAULT FALSE,
  version          INT NOT NULL DEFAULT 1,
  is_current       BOOLEAN NOT NULL DEFAULT TRUE,
  last_probe_at    TIMESTAMPTZ,
  last_probe_ok    BOOLEAN
);
CREATE UNIQUE INDEX mcp_official_slug_current ON mcp_servers (slug)         WHERE scope='official' AND is_current;
CREATE UNIQUE INDEX mcp_user_slug_current     ON mcp_servers (user_id,slug) WHERE scope='user'     AND is_current;

CREATE TABLE cron_jobs (
  id            UUID PRIMARY KEY,
  scope         TEXT NOT NULL CHECK (scope IN ('official','user')),
  user_id       UUID REFERENCES users(id) ON DELETE CASCADE,
  slug          TEXT NOT NULL,
  cron_expr     TEXT NOT NULL,
  prompt        TEXT NOT NULL,
  webhook_url   TEXT,
  enabled       BOOLEAN NOT NULL DEFAULT TRUE,
  mandatory     BOOLEAN NOT NULL DEFAULT FALSE,
  last_fired_at TIMESTAMPTZ,
  strikes       INT NOT NULL DEFAULT 0
);

CREATE TABLE user_official_opt_outs (
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  entity  TEXT NOT NULL CHECK (entity IN ('skill','instruction','command','mcp_server','cron_job')),
  slug    TEXT NOT NULL,
  PRIMARY KEY (user_id, entity, slug)
);

-- ============ Per-user state ============
CREATE TABLE user_documents (
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  kind        TEXT NOT NULL CHECK (kind IN ('memory.md','user.md')),
  content     TEXT NOT NULL DEFAULT '',
  version     INT NOT NULL DEFAULT 1,
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, kind)
);

-- ============ Sessions / messages ============
CREATE TABLE sessions (
  id              TEXT PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  source          TEXT NOT NULL,
  model           TEXT,
  started_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  ended_at        TIMESTAMPTZ,
  end_reason      TEXT,
  message_count   INT NOT NULL DEFAULT 0,
  tool_call_count INT NOT NULL DEFAULT 0,
  input_tokens    BIGINT NOT NULL DEFAULT 0,
  output_tokens   BIGINT NOT NULL DEFAULT 0,
  title           TEXT
);
CREATE INDEX sessions_user_started ON sessions (user_id, started_at DESC);

CREATE TABLE messages (
  id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  session_id    TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  role          TEXT NOT NULL,
  content       TEXT,
  tool_call_id  TEXT,
  tool_calls    JSONB,
  tool_name     TEXT,
  ts            TIMESTAMPTZ NOT NULL DEFAULT now(),
  tsv           tsvector GENERATED ALWAYS AS
                (to_tsvector('simple', coalesce(content, ''))) STORED
);
CREATE INDEX messages_user_session_ts ON messages (user_id, session_id, ts, id);
CREATE INDEX messages_tsv             ON messages USING GIN (tsv);

-- ============ Skill evolution ============
CREATE TABLE skill_proposals (
  id              UUID PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  skill_slug      TEXT NOT NULL,
  base_version    INT,
  body            TEXT NOT NULL,
  fitness         JSONB,
  status          TEXT NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending','approved','rejected','superseded')),
  s3_artefact_key TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  decided_at      TIMESTAMPTZ
);

-- ============ Audit & metering ============
CREATE TABLE audit_log (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  ts          TIMESTAMPTZ NOT NULL DEFAULT now(),
  actor_id    UUID NOT NULL REFERENCES users(id),
  actor_role  TEXT NOT NULL,
  action      TEXT NOT NULL,
  target      JSONB NOT NULL,
  diff        JSONB,
  request_id  UUID
);
CREATE INDEX audit_log_actor_ts  ON audit_log (actor_id, ts DESC);
CREATE INDEX audit_log_action_ts ON audit_log (action, ts DESC);

CREATE TABLE usage_events (
  id                  BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id             UUID NOT NULL REFERENCES users(id),
  session_id          TEXT,
  ts                  TIMESTAMPTZ NOT NULL DEFAULT now(),
  model               TEXT NOT NULL,
  input_tokens        INT NOT NULL,
  output_tokens       INT NOT NULL,
  cached_input_tokens INT NOT NULL DEFAULT 0,
```

### Phase 35 — Streaming + permissions + idempotency (+800 / +500 LOC)

**Objective.** Ship the chat surface end-to-end. SSE streaming wraps
the existing `run_turn_streaming`; ask-mode emits `permission_request`
events resolved by a separate decisions endpoint; idempotency keys
deduplicate retries.

**Deliverables.**

- `src/lite_horse/web/sse.py` — SSE event encoder; helpers
  `event_delta`, `event_tool_call`, etc.
- `src/lite_horse/web/routes/turns.py` — `POST /v1/turns`,
  `POST /v1/turns:stream`, `POST /v1/turns/{turn_id}/decisions`,
  `POST /v1/turns/{turn_id}:abort`.
- Idempotency: middleware reads `Idempotency-Key` header on `POST
  /v1/turns*`, stores `{user_id, key} → turn_id` in Redis with 24 h TTL.
  Replay returns the cached response.
- Ask-mode: `RequestContext.permission_mode='ask'` causes the agent
  hook to publish a Redis pub/sub message and await the decision.
  Decisions endpoint resolves the future. Tool execution proceeds or
  raises `ToolDenied` based on the answer. SSE stream emits
  `permission_request` and pauses until decided.
- Per-session distributed lock via `RedisSessionLock` with TTL 5 min,
  acquired in `turns` handler, released in finally.
- `tests/web/test_turns_stream.py` — fake LLM emits a deterministic
  stream; client receives SSE events in expected order.
- `tests/web/test_idempotency.py` — duplicate request with same key
  returns identical response without invoking the agent twice.
- `tests/web/test_ask_mode.py` — turn pauses on `permission_request`,
  decision unblocks, tool runs.

**Acceptance.**

- A streaming turn delivers `delta` events as the LLM streams,
  `tool_call` and `tool_output` per tool, `usage` near the end, `done`
  to close.
- Two concurrent `POST /v1/turns` for the same `session_key` from two
  ECS tasks serialise via Redis lock; the second waits, then proceeds.
- Idempotency-Key replay returns cached SSE bytes (or the cached
  non-streaming response) without a new LLM call.
- Ask-mode pauses at the first write tool, resumes on `allow`,
  errors out on `deny`.
- Abort endpoint cancels the in-flight turn within 1 s of receipt.

### Phase 36 — Scheduler + worker services, org-wide cron (+800 / +500 LOC)

**Objective.** Move scheduling out of the api process. One scheduler

**Deliverables.**

  one message per active user for official-scope jobs).
- `src/lite_horse/worker/__main__.py` — long-running async loop

  `run_turn` used by the api service. Reuses repositories, layered
  config, providers — same code, different entry.
- `src/lite_horse/cron/scheduler.py` refactored to expose

- `cron/delivery.py` — webhook fan-out preserved; signed with the

- `Dockerfile` produces three command modes: `uvicorn …` (api),
  `python -m lite_horse.scheduler` (scheduler), `python -m
  lite_horse.worker` (worker). Same image, different `command:`.
- ECS task definitions for `scheduler` (desiredCount=1) and `worker`

- `tests/scheduler/test_fanout.py` — fakes `time.now`, asserts fanout

**Acceptance.**

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Async port introduces transaction-boundary regressions | Keep a SQLite-backed `LocalSessionRepo` impl behind the same Protocol; parity tests gate the cutover (Phase 32) |
| RLS misconfiguration leaks data | Phase 39 leak test in CI; do not enable RLS in unit tests so missing `WHERE user_id=` bugs are still caught |
| RDS connection storms during deploys | Pool ceiling × 2 ≤ `max_connections`; switch to RDS Proxy when crossing 50 ECS tasks |
| Bad mandatory official instruction breaks everyone | Versioning + rollback; admin UI shows preview-against-test-prompt before publish (frontend concern) |
| Cost runaway from BYO-key misconfig | Per-user daily cost budget enforced in Phase 39 |
| `tsvector` tokenisation differs subtly from FTS5 | Phase 32 parity corpus gates the cutover at ≥95% |
| Scheduler service is a single point of failure | One task is acceptable; APScheduler resumes from PG jobstore on restart; ECS auto-restarts dead tasks |
| Local MinIO drift from real S3 behaviour | Integration tests run against both moto (in-process) and MinIO (docker); production smoke gated on staging deploy |
| KMS rate limits | DataKey caching via AWS Encryption SDK (5 min, 1000 entries) per Phase 31 |

   route + token refresh; Phase 31 only reserves the column shape.

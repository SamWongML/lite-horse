# HTTP API

The v0.4 cloud surface is a multi-tenant FastAPI service. Every request
carries a JWT minted by the parent webapp; `sub` becomes the lite-horse
`user_id` and is the sole tenancy boundary (Postgres RLS + Redis key
namespacing both pivot on it).

> **OpenAPI is the source of truth.** This document describes the surface
> shape, authentication contract, and operational semantics. The schemas
> for individual routes are generated by FastAPI — fetch them at
> `GET /openapi.json` from a running instance, or import the FastAPI app
> via `from lite_horse.web.app import create_app` and call
> `create_app().openapi()` to render them offline.

`docs/EMBEDDING.md` describes the **deprecated** v0.2/v0.3 embedded
Python surface. New integrations should use the HTTP API.

---

## Authentication

* Bearer JWT in `Authorization: Bearer <token>`.
* Issuer + JWKS URL come from `LITEHORSE_JWT_JWKS_URL` (cached for 1 h).
* `aud` must match `LITEHORSE_JWT_AUDIENCE`.
* `sub` is the lite-horse `user_id`. It is opaque; the webapp owns the
  mapping back to its own user identity.

A request without a valid token is rejected with `401`. A request whose
`sub` is not yet provisioned in `users` is auto-onboarded on first call.

## Tenancy isolation

* Every DB transaction calls `SET LOCAL app.user_id = '<sub>'`. RLS
  policies on `sessions`, `messages`, `user_documents`, `skill_proposals`,
  and `usage_events` reject reads/writes targeting any other tenant.
* Phase 39's `tests/security/test_rls_leak.py` runs nightly in staging
  to verify no leaks.
* Redis keys are user-scoped (`rate:turn:{user_id}:...`,
  `cost:day:{user_id}:...`, `idem:{user_id}:...`).

## Top-level surface

| Group                         | Path prefix              | Auth        |
|-------------------------------|--------------------------|-------------|
| Turns (chat)                  | `/v1/turns*`             | user JWT    |
| User config (skills, MCP, …)  | `/v1/users/me/*`         | user JWT    |
| Admin (official-scope CRUD)   | `/v1/admin/*`            | admin JWT   |
| Ops (health, version, debug)  | `/v1/ops/*`              | unauth/health |

### Turns

* `POST /v1/turns` — non-streaming JSON. Returns final assistant text +
  usage + cost in micro-USD.
* `POST /v1/turns:stream` — SSE; one `data: {...}` line per event,
  terminated by an `event: done`. Replayable via `Idempotency-Key`
  (24 h Redis cache; the SSE byte stream is replayed verbatim).
* `POST /v1/turns/{turn_id}/decisions` — answer an in-flight ask-mode
  permission prompt. Routed through `PermissionBroker` so the running
  task on any ECS node receives the decision.
* `POST /v1/turns/{turn_id}:abort` — cancel an in-flight turn.

#### Headers

| Header              | Required          | Effect |
|---------------------|-------------------|--------|
| `Authorization`     | yes               | Bearer JWT |
| `Idempotency-Key`   | recommended       | 24 h dedupe per `(user_id, key)` |
| `X-Request-Id`      | no                | Echoed back; otherwise auto-assigned |

#### Limits

* **Rate limit:** 60 turns/min per user (override via
  `users.rate_limit_per_min`). Excess returns `429 RATE_LIMIT`.
* **Daily cost budget:** opt-in per user via
  `users.cost_budget_usd_micro`. Alerts fire at 80%; 100% returns
  `403 cost_budget_exceeded` until the next UTC day.

### User config

Every CRUD route under `/v1/users/me/*` operates on the calling user's
scope only — the URL never carries a user id. Resources:

* `skills/*` — Markdown skill files; activated heuristically per turn.
* `mcp-servers/*` — external MCP server registrations. Auth headers
  are KMS-encrypted at rest under `EncryptionContext={"user_id": ...}`.
* `commands/*` — slash-command shortcuts.
* `cron-jobs/*` — scheduled jobs delivered via signed webhook.
* `byo-keys/*` — BYO OpenAI / Anthropic / GitHub credentials.
* `settings` — rate limit, cost budget, default provider.
* `:request-delete` / `:cancel-delete` — GDPR account deletion.

#### GDPR deletion

* `POST /v1/users/me:request-delete` schedules a deletion at
  `now() + interval '7 days'` and returns a
  `GdprDeleteRequestOut` (`requested_at`, `scheduled_at`,
  `cancelled_at`, `completed_at`). A second open request returns
  `409 gdpr_delete_already_pending`.
* `DELETE /v1/users/me:cancel-delete` cancels the open request if
  before `scheduled_at` (returns the updated row); `404` if no
  open request exists.
* When `scheduled_at` is reached, the daily `gdpr_delete` worker
  exports every tenant-scoped row to
  `s3://$LITEHORSE_S3_BUCKET_AUDIT_ARCHIVE/gdpr/<request_id>.jsonl`,
  deletes from every tenant table in one transaction, tombstones
  `audit_log.actor_id` (FK is `ON DELETE SET NULL` so audit
  history survives), and stamps `completed_at` + `archive_s3_key`.

### Admin

`/v1/admin/*` is the official-scope surface. Each entity is versioned;
every write produces an `audit_logs` row and publishes
`effective-config-invalidate` so api tasks evict their caches within
~1 s. Mandatory entries cannot be opted out by users (422).

### Ops

* `GET /v1/ops/health` — liveness, returns 200 unless DB or Redis are
  unreachable.
* `GET /v1/ops/version` — package version + git sha.
* `GET /v1/ops/debug/effective-config` — local-env only.

## Idempotency

`POST /v1/turns` and `POST /v1/turns:stream` cache the full response
keyed by `(user_id, Idempotency-Key)` for 24 h. Streaming caches the
exact bytes emitted so a replay produces an identical SSE byte stream.
A retry with the same key but a different request body returns
`409 idempotency_key_conflict`.

## Error envelope

All non-2xx responses carry:

```json
{
  "error": {
    "code": "RATE_LIMIT" | "FORBIDDEN" | "INTERNAL" | ...,
    "message": "human readable",
    "request_id": "..."
  }
}
```

Codes map to `ErrorKind` in `src/lite_horse/web/errors.py`. The
classifier is shared with the embedded path so error semantics are
identical across surfaces.

## Cron + worker

The api process does not run cron. A standalone `scheduler` service
ticks every 60 s, expanding official-scope jobs into per-user
`CronMessage` records. Daily ticks fan out `EvolveMessage` (users
with ≥10 trajectories since the last evolve), `EvolveGepaMessage`
(skills with `gepa: true` frontmatter), `GdprDeleteMessage` (due
deletion requests), `AuditShipMessage` (rows older than 90 days),
and `CurateMessage` (one per `(user, agent)` slice owning any
skill); the hourly `summarize` tick scans idle sessions. The
`worker` service long-polls SQS; for `cron` payloads it runs the
turn and signs the webhook delivery (HMAC-SHA256 with
`LITEHORSE_WEBHOOK_SECRET`); for `evolve` / `evolve_gepa` it
produces a `pending` `skill_proposal` row for human review;
for `gdpr_delete` it exports + purges + tombstones; for
`audit_ship` it streams aged `audit_log` rows as JSONL to the
audit-archive bucket and `DELETE`s them from PG.

## Security headers

Cloud responses carry HSTS
(`max-age=31536000; includeSubDomains; preload`), CSP
(`default-src 'none'`), `X-Frame-Options: DENY`, and
`Referrer-Policy: no-referrer` via
`web/middleware/security_headers.py`. The middleware is
no-op when `LITEHORSE_ENV=local` so the CLI-served `/debug/*`
surface stays unconstrained.

## Observability

Every response carries `X-Request-Id`. Logs are JSON via structlog
(`request_id` / `user_id` / `session_key` / `turn_id` bound on a
contextvar). Traces are OTLP/HTTP exported when
`OTEL_EXPORTER_OTLP_ENDPOINT` is set. EMF metrics are emitted as JSON
log lines and aggregated by the CloudWatch agent —
`turns_total`, `tokens_total`, `cost_usd_micro`, `errors_total`,
`http_request_duration_ms`, `cron_fires_total`,
`evolve_enqueued_total`.
